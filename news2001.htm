August 18, 2001

MAMED beta 4 is ready and available on the downloads page. It's based on the fairly up to date mame37b15 source code base. The latest mame core removes 8-bit support so I probably will not update to that, because some games will run out of memory in 16-bit mode. The 68K cpu emulation should be fixed however some sound drivers still appear to not work, outzone for one. Be sure to turn off sound in those games or memory corruption could occur. Hopefully I can fix it next time around. There is a memory gauge built into the options menu so you can see how much memory is being used for certain games. Memory is very tight in some larger games, some will not even run if you use hyper's menu and the pack must be run directly from kosh or boot.rc, outzone again is one of those. I modified hyper's menu for the extra pack 16 & neomamed, the menu will also wrap now. You can also exit back to hyper's menu from each pack, this cleans up any memory fragmentation that has occured while running a particular pack. There is also an option to hide clones which is nice to clean up the game list a little. Fake sound also works now. Many games that didn't work before should work now. Also four controllers are supported the first one controls the menus, the buttons changed a bit because the joysticks are supported as true joysticks (with analog support) instead of mapping to the keyboard, you still don't need a keyboard to play. Have fun.

August 11, 2001

Just got back from the Mall of America store and got to see the grand opening of the Apple Computer store. Gotta get out of the house sometimes. Also a big congradulations to the makers of the Delicious DC Demo for winning first prize at the Assembly 2001. It really is impressive.

A little update to keep everyone happy. Food fight is running along with neogeo games joyjoy & puzzled which means 68K stuff is working, a little slowly right now as I have MAMEDBG turned on. I have more work to do though, the 68K cpu instruction cycles table is not correct for some reason (lot's of zero's and big numbers so I hacked it to always be 10 cycles per instruction for the moment), I don't know if it's getting trashed or isn't created correctly in the first place, I'm investigating. I have identified about 10 blocks (32K) of memory that are not getting freed after running a game, which leads to memory fragmentation, really need to track that down where those occur so it doesn't stop working after playing 10 games or so. There seem to be some random occurances of memory corruption as well that show up more frequently when I put printf() in the code, could be a KOS thing or something in my stuff. I did track down one memory corruption thing occuring in my palette table code which I fixed. If there are any other CPUs that exhibit the same behavior as the 68K did, black screen but menus work and you can return to the game list, I'd like to hear about it.

Also about memory use, after running on one of the above neogeo games the maximum total memory allocation was 7,571,216 bytes. To run just pacman required 2,276,748 bytes. This is in addition to KOS and the MAMED application code. Just to give everyone an idea of the memory overhead for display buffers, graphic tiles, roms, cpu ram, cpu tables, etc. required to run a MAME game. So even if the rom is only 9MB uncompressed doesn't mean it's going to run, there are a lot of other things that require RAM. In the future I might be able to swap ROM's in an out of RAM and allow loading of much larger games but for the moment don't expect really large games to run. Just letting you know so you don't get your hopes up too high.

Oh, and one more thing, outzone runs really well for a little while but no sound yet.

MAMED changes

Fixed floating point calculations, fixes some sound emulation and vector games and background png image loading. Games which use the 68K are marked with all CAPS, cause the 68K cpu isn't in this release (saves about 1MB or RAM). If you select a game and it doesn't run you selected a game that uses 68K, you ran out of RAM, you don't have the correct rom set for MAME core 37b6, you selected a clone rom set and you don't have the parent set on the same CD. If a game doesn't have correct sound you might need samples, some games that don't require samples now did require them for this older version of the MAME core. I now support both 8 & 16 bit graphics modes. In 16 bit, if possible I avoid using lookup tables and use direct copy which is much faster (BTW the compiler lib built-in memcpy() is way slow, avoid in your own stuff if possible). Use 8-bit for larger games that require more memory. I also added an new graphics mode that tries to scale a little better. You can run MAMED from the command line with a rom name argument to create a self boot single game CD if you wish. Rescanning of the CD is only done when you select "RESCAN". You can put a snap directory on the CD and the png file corresponding to the rom name is displayed in the background in the game list menu (looks really nice BTW). There is preliminary support for a config file on the CD called MAME.CFG you can use it if you like. Overall stability should be improved and many more games that didn't run before should run now. If you screw up your boot disk but do manage to get KOS on there I will run KOSH which is built into the KOS image in /rd. And finally the X & B buttons are swapped from the last release.

Know MAMED problems

68K emulation doesn't work, duh? Sometimes sound startup and shutdown is not in sync with the HW and can cause the Dreamcast to reset, I'm looking into it, can cause some games to fail at startup or shutdown at random. There are a few games that should work that don't, playchoice10 comes to mind, I'm working on it. Cheat, History, Hiscore, Mameinfo are not working, don't even try and put the file on the CD. Memory fragmentation needs to be improved to avoid crashes and allow switching to other packs.

KOS changes

Increased the iso9660 disk cache from 32K to 128K (should hold any size rom disk you stick in, I might reduce it a bit in the future but I have room for the moment). Memory allocation that fails no longer reset the dreamcast (the MAME source code is pretty good and can recover correctly from out of memory conditions). Memory can still get fragmented after using MAMED for a while and cause out of memory conditions and even crashes. I also adjusted the stack ceiling from 64K to 128K to avoid stack overflow problems.

July 18, 2001

The CD swap bug has been fixed. Turned out to be a single character in the KOS iso9660 driver (uint32 changed to int32) for those that want to know. So now you need my special version of KOS to run MAMED on your dreamcast, although this version should run any standard KOS application without problems. I've included the standard KOS boot script support in this special version so you can change which mamed.klf gets run of a CD or so you can run some kind of KOS menu application to select one. Disk swapping should not cause any problems, to refresh the screen just try and load something that's not on the disk by pressing start from the game list and it will rescan the CD correctly. You can download the plain files here. Just be sure to edit the boot.rc file to run the version of mamed.klf you want. I've also embedded the stream.drv in the /rd inside KOS so you don't need it on the CD anymore.

OK now on to the MAMED packs. All cpu and sound drivers are inside in each pack so they will be too big to load onto /rd and run it from there so they must be on a CD to run preferably the CD with KOS above. All MAME drivers are included and separated by manufacture groups but not all will run because of memory constraints. I tried to make all the packs about the same size (~5MB). I can fairly easily make special builds to run a particular game with only the bare minimum of code but I will work on that later. Remember the uncompressed rom size is important but also each driver has to emulate ram chips inside the machine too so don't expect a rom that uncompresses to 14.6MB to run on the dreamcast you need space for ram chips and the emulator code and even then it might be just too slow. Also remember this is based on MAME v37b6 not the latest and some rom have changed.

These are big files so don't just go downloading all of them at once. Download one with the KOS above and give it a try first. Please let me know if they work, the last beta build was diffcult for most people. This one should be much easier to get working. I'll get around to packaging up the source when I get a chance and hopefully others can make some selfboot images, I'm no good at it with my CDR. Remember this is still beta and there will be problems running some games. Enjoy.

Update: crjack says he has burned a working MAMED self boot image so it looks like its a good build this time, everyone start burning.
